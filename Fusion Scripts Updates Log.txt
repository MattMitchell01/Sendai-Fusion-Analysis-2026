  PART A — Extract Traces From Video
  ─────────────────────────────────────────────────────────────────────────────

  (1)  FFT XY-Drift Correction Gets an Ordered List of Backup Regions
       TYPE:   Method Change / Robustness
       WHY:    A single fixed ROI for the FFT cross-correlation (see February 2026,
               Part A, item 1) can fail on some videos -- e.g. if that region happens
               to sit over an area of the image where there is a disturbance. 
               Fast_Fourier_Transform is now tried against an ORDERED list of regions
               (Options.FFTRegionList): the middle 40% of the frame first, then all
               four corners as backups. After each region's XY-shift graph is plotted,
               the user approves or rejects it; on rejection, the next region in the
               list is tried automatically. Only once every region has been rejected
               does the program fall back to Single_Particle_Tracking (manual click,
               re-tried until approved). This makes the FFT method self-recovering
               instead of a single-shot pass/fail.
       FILES:  Setup_Options_Extract_Traces.m   (Options.FFTRegionList: 5 regions)
               Find_And_Analyze_Particles.m     (per-region try/approve loop)
               Fast_Fourier_Transform.m         (takes a Region argument)

  (2)  Output Struct Fields Finalized and Fully Documented
       TYPE:   Method Change / Documentation
       WHY:    VirusDataToSave (per-virus) and OtherDataToSave (per-video) are the

  PART B — Lipid Mixing Trace Analysis  (NEW CLASSIFIER — full rewrite)
  ─────────────────────────────────────────────────────────────────────────────

  (1)  Part B Rebuilt Around Explicit Fusion/Unbound Scoring
       TYPE:   Method Change / Full Rewrite
       WHY:    Bob's original Start_Trace_Analysis / Analyze_Current_Data_Set used
               gradient-cluster heuristics that were hard to reason about and tune.
               The new pipeline scores every frame of a trace with two transparent,
               independently-tunable detectors and assigns a designation from their
               cluster counts. Each detector is a set of binary tests (bits) so a
               near-miss is legible: the score itself tells you which test failed.
                 · Seek_Unbounds  — 4 backward-looking tests, score 15 = unbound frame
                 · Seek_Fusion    — 5 forward-looking tests, score 31 = fusion frame
               Find_Fusion_Frame / Find_Unbound_Frame then localize the exact event
               frame with a step-function fit. Classify_Trace turns cluster counts into
               '1 Fuse' / '2 Fuse' / 'Unbound' / 'No Fusion' / 'Other'.
       FILES:  Analyze_Trace_Data.m            (new core pipeline)
               Seek_Fusion.m / Parameters_Seek_Fusion.m
               Seek_Unbounds.m / Parameters_Seek_Unbounds.m
               Find_Fusion_Frame.m / Find_Unbound_Frame.m
               Classify_Trace.m
               Calculate_Sliding_Window.m
               Setup_Options.m
               Start_Trace_Analysis.m          (adapted entry point)

  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─

  PART R — Review Old Data  (ENTIRELY NEW SECTION)
  ─────────────────────────────────────────────────────────────────────────────

  (1)  Reprocess Legacy Datasets Through the Current Pipeline
       TYPE:   New Feature / New Section
       WHY:    Old datasets analyzed and hand-reviewed under Bob's original pipeline are
               a large, already-ground-truthed pool for grading the new Part B. Part R
               reshapes legacy input into the format current Part B expects, runs the
               UNMODIFIED Part B and Part C, and flags every trace where the new
               algorithm disagrees with the old human review — so a reviewer only looks
               at the disagreements, sorted into the normal Low/Medium/High + H1–H5

               validates the legacy frame-indexing convention on every file before
               converting. Part C gained two additive, default-off options
               (ReviewOldDataMode, ReviewOldDataOnlyDifferences) to drive these sessions
               without touching the live pipeline.

  PART A — Extract Traces From Video
  ─────────────────────────────────────────────────────────────────────────────

  (1)  New Method to Determine XY Sample-Wide Offset — Fast Fourier Transform
       TYPE:   Method Change
       WHY:    FFT provides a more accurate and precise way to measure
               sample-wide pixel offsets across the timelapse. Single-particle
               tracking (SPT) is inherently noisier because centroid positions
               are highly sensitive to local intensity fluctuations, shape
               changes, and background variation, whereas FFT leverages
               information from an entire image region. SPT also fails when the
               tracked particle is lost, forcing a restart.
       FILES:  Find_And_Analyze_Particles.m   ln 45–94
               Fast_Fourier_Transform.m       (new function)
               Single_Particle_Tracking.m     (renamed)

  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─

  (2)  Shrinking ROIs to Better Match Virus Size (for Synthetic Vesicle ppl)
       TYPE:   Method Change
       WHY:    The original ROIs were significantly larger than the viruses
               themselves (e.g., a 9×9 ROI for a 4-pixel virus). ROI sizing was
               modified so that the side length is determined by sqrt(AreaForROI)
               rather than sqrt(2 × AreaForROI). This produces tighter ROIs that
               scale more appropriately with virus size and improves the
               signal-to-noise ratio of the extracted traces. Residual
               frame-to-frame noise appears to arise from inherent intensity
               fluctuations of the virus pixels themselves.
CRITICAL NOTE: This change is NOT for Fusion to RBCs. Why? --> we observe virions 
	       spinning in circles and moving ever so slightly on the RBC 
	       tethered vesicle surface. The oversized ROIs actually solve this
	       issue. 
	FILES:  Simplified_Test_Goodness.m   ln 51

================================================================================
  NOTES & IMPORTANT FINDINGS
================================================================================

  (1)  Determining Frame-to-Frame XY Drift
       VERDICT:  New Method — Fast Fourier Transform (FFT)

       FFT is a superior method to single-particle tracking for determining the
       XY offset of the sample across frames of the timelapse. See Part A
       update (1), February 2026 for full details.

  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─

  (2)  Background Subtraction
       VERDICT:  Keep using RoughBackground

       The RoughBackground method — mean(median(FrameImage)) — works much
       better than initially expected. Frame-to-frame variation scales to
       2–10 intensity units per ROI, which is negligible relative to particle
       intensity values of 0–10000. Top Hat subtraction was also tested (see
       Part A update (1), June 2026) and found not to improve upon this.

  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─  ─

  (3)  Size of Virus ROI
       VERDICT:  Shrink the ROIs

       Previously, nearly all ROIs were 9×9 (81 pixels). After the adjustments
       in Part A update (2), most ROIs are now 5×5, while larger viruses retain
       proportionally larger ROIs. It appears 5×5 is the minimum ROI produced
       by the current approach — MATLAB's bwconncomp and regionprops effectively
       impose a minimum area of ~25 pixels. Further modifications to
       Simplified_Test_Goodness.m could allow ROIs to shrink below this limit.
